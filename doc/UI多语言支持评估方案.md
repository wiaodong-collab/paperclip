# UI 多语言支持评估方案

日期：2026-03-25
范围：仅评估 `ui/` 前端界面的多语言支持，不覆盖 `server/`、`cli/`、数据库存量文本、模型输出内容或文档站点。

## 1. 目标

为 Paperclip 的 `ui/` 提供可持续维护的多语言能力，使界面文案可以按语言切换，并为后续逐步扩展更多语言保留清晰的工程边界。

本评估文档不仅说明当前现状，还给出推荐方案和分阶段落地建议，供后续实施使用。

## 2. 当前现状

基于当前仓库状态，`ui/` 尚未具备系统化的 i18n 能力：

- 没有发现 `i18next`、`react-i18next`、`react-intl`、`next-intl` 等常见 i18n 依赖。
- UI 入口当前只接入了路由、主题、查询、Toast 等 Provider，没有国际化 Provider。
- 页面和组件中存在大量直接写死的英文字符串。
- 没有发现语言资源目录，例如 `locales/`、`messages/`、`translations/`。
- 当前代码中出现的 `localeCompare` 或局部格式化逻辑，仅用于排序或显示，不构成完整多语言方案。

这意味着当前 UI 的文案体系是“硬编码英文为主”，不支持：

- 按语言切换界面文案
- 统一管理公共文案
- 变量插值、复数规则、命名空间拆分
- 渐进式扩展第二语言或更多语言

## 3. 约束与边界

本次评估只讨论 UI 多语言，不把问题扩大到整个产品栈。

明确不在本次方案内的内容：

- 后端 API 返回错误消息的多语言化
- 用户自己输入的内容翻译
- Agent 输出内容翻译
- 数据库中已存在业务文本的翻译或迁移
- 文档站点和 CLI 文案国际化

本次推荐方案的目标是：先把前端界面文案纳入统一国际化体系，再允许按页面逐步迁移，不要求一次性替换全部硬编码文本。

## 4. 可选方案对比

### 方案 A：轻量自研国际化层

做法：

- 在 `ui/` 内自行实现 `LocaleProvider`
- 用 TypeScript 对象或 JSON 保存文案字典
- 提供简单的 `t(key)` 方法读取文案

优点：

- 依赖最少
- 初始接入速度快
- 对极小型项目足够直接

缺点：

- 变量插值、复数规则、回退语言、命名空间、懒加载等能力都要自行维护
- 后续扩展英文以外语言时，工程复杂度会快速上升
- 容易演变为“早期简单、后期难以维护”的半成品方案

适用条件：

- 文案规模长期很小
- 只需要少量页面双语
- 团队明确接受后续自行维护国际化基础设施

结论：

- 不推荐作为 Paperclip UI 的长期方案

### 方案 B：`i18next + react-i18next`

做法：

- 引入成熟 i18n 方案作为 UI 统一国际化基础设施
- 在 React 层通过 Provider 和 Hook 使用文案
- 按命名空间拆分语言资源

优点：

- 能力完整，适合持续演进
- 社区成熟，文档和实践多
- 支持回退语言、变量插值、复数、命名空间、语言检测
- 对 React 项目接入自然，适合页面逐步替换硬编码字符串
- 便于把全局公共文案和页面文案分层管理

缺点：

- 初始接入成本高于自研
- 需要统一约束 key 命名和资源组织，否则后期也会混乱

适用条件：

- UI 还会继续扩展
- 希望先建立正确基础设施，再逐步迁移文案
- 需要兼顾短期可落地与长期维护成本

结论：

- 推荐

### 方案 C：`react-intl`

做法：

- 使用 `react-intl` 进行消息定义、格式化和语言切换

优点：

- 日期、数字、消息格式化能力强
- 消息定义体系较清晰

缺点：

- 对当前项目这种“先把散落硬编码文案系统化抽离”的阶段，不如 `react-i18next` 灵活
- 从现有代码平滑迁移的工程体验通常不如 `i18next` 生态顺手

适用条件：

- 团队非常强调 ICU Message 格式
- 文案格式化复杂度高于普通页面文案管理

结论：

- 可行，但不是当前 UI 的最优选择

## 5. 推荐方案

推荐采用：`i18next + react-i18next`

推荐理由：

1. 当前 UI 还没有任何国际化基础设施，优先级不是“写一个最轻的临时方案”，而是“建立一个后续可持续扩展的标准方案”。
2. 当前问题的核心不是格式化能力，而是页面和组件中存在大量散落硬编码文案，需要一个能支持渐进迁移的体系。
3. `react-i18next` 更适合 React 项目的页面级改造，可以先接入框架，再分模块替换硬编码文本，而不需要一次性重写所有组件。
4. 后续如果要做语言切换、浏览器语言检测、按页面拆分语言包、统一测试，这套方案都更稳。

## 6. 推荐架构

### 6.1 总体原则

- 默认语言保持当前英文，避免现有体验突然变化
- 增加简体中文作为第一种新增语言
- 所有新增 UI 文案必须进入 i18n 资源，不再直接写死字符串
- 旧代码允许分阶段迁移，但新代码不能继续增加硬编码文案

### 6.2 资源组织

建议在 `ui/src` 下建立独立 i18n 目录，语言资源按“语言 + 命名空间”组织。

建议结构示例：

```text
ui/src/i18n/
  index.ts
  config.ts
  locales/
    en/
      common.json
      agents.json
      issues.json
      projects.json
      goals.json
      settings.json
    zh-CN/
      common.json
      agents.json
      issues.json
      projects.json
      goals.json
      settings.json
```

组织原则：

- `common` 放按钮、导航、状态、通用错误、空状态等跨页面文案
- 业务页面按领域拆分，避免一个超大资源文件
- key 使用语义命名，不使用整句英文作为 key

示例：

```json
{
  "pageTitle": "Agents",
  "empty.noCompanySelected": "Select a company to view agents."
}
```

而不是：

```json
{
  "Select a company to view agents.": "Select a company to view agents."
}
```

### 6.3 运行时接入

建议在 UI 启动入口挂载 i18n Provider，并通过 Hook 在组件中取文案。

设计要点：

- 在 `ui/src/main.tsx` 的现有 Provider 栈中接入 i18n 初始化
- 页面组件统一通过 `useTranslation()` 或项目内二次封装的 `t()` 读取文案
- 所有公共状态标签、按钮文案、空状态文案逐步统一收口

### 6.4 默认语言与切换策略

建议顺序：

1. 默认语言：英文
2. 初始语言来源：本地持久化设置优先，其次浏览器语言检测，最后回退到英文
3. 提供显式语言切换入口，但不要求第一阶段一定上线复杂的语言设置页

这样做的好处：

- 不会破坏现有默认英文体验
- 为中文用户提供明确切换路径
- 不强依赖后端参与

### 6.5 插值、复数和格式化

虽然第一阶段不必把所有复杂场景都做完，但架构需要从一开始支持：

- 变量插值，例如用户名、数量、模型名
- 复数规则
- 日期时间和数字格式化

注意：

- 文案翻译和时间数字格式化是两类问题，不能混在各组件里随意处理
- 动态文本应优先通过统一工具层处理，而不是每个页面各写一套格式化逻辑

## 7. 迁移策略

推荐采用“渐进替换”，而不是“一次性全量替换”。

### 阶段 1：建立基础设施

目标：

- 接入 `i18next + react-i18next`
- 建立语言资源目录和命名规范
- 接好默认语言、回退语言和本地存储
- 明确新增文案必须走 i18n

产出：

- UI 已具备双语基础能力
- 后续页面改造可以并行推进

### 阶段 2：抽取全局公共文案

优先抽取：

- 顶层导航
- 常用按钮
- 通用空状态
- 通用状态标签
- 常见错误提示

原因：

- 复用率高
- 替换后收益立刻可见
- 能先沉淀 key 设计规则

### 阶段 3：按页面逐步迁移

建议优先顺序：

1. 高频主页面
2. 创建/编辑类表单
3. 错误反馈和 Toast
4. 次级页面与实验性页面

原因：

- 高频页面最影响用户感知
- 表单和提示文案最适合统一规范

### 阶段 4：补测试与治理规则

建议补充：

- i18n 初始化测试
- 核心页面渲染测试
- 关键 key 缺失的检测机制
- 对新增硬编码字符串的约束检查

目的：

- 防止新页面绕过 i18n 直接继续写死文案
- 防止语言资源缺 key 时静默退化

## 8. 风险与注意点

### 8.1 硬编码文案分散，初次治理触达面广

当前问题不是缺一个库，而是缺文案治理边界。接入库本身很快，但真正的工作量在于把分散在页面和组件中的硬编码字符串逐步收口。

### 8.2 key 命名如果无规则，后期会变成新的混乱源

如果没有命名约定，后面会出现：

- 同义 key 重复
- 页面间文案复用关系不清晰
- 翻译资源越来越难维护

因此需要尽早约定：

- 公共文案进 `common`
- 页面文案按业务域拆分
- key 用语义命名，不引用具体英文原句

### 8.3 状态文案、枚举标签、错误提示需要统一来源

很多页面都可能展示相同状态值，例如 agent 状态、issue 状态、按钮动作、空状态。这些内容不应在每个页面重复翻译，而应统一通过 shared mapping 或 UI 文案层输出。

### 8.4 不应把“语言切换”与“内容翻译”混为一谈

UI i18n 只负责界面文案，不负责：

- 用户内容翻译
- Agent 输出翻译
- 历史数据翻译

如果不明确边界，后续会把问题扩展成更大的产品需求，增加实施风险。

## 9. 推荐落地范围

如果后续启动实施，建议第一批只覆盖这些内容：

- 顶层导航与页面标题
- 通用按钮与表单标签
- 空状态和无数据提示
- 常见客户端错误提示
- 主要列表页和详情页中的固定 UI 文案

暂缓内容：

- 富文本编辑器内的复杂交互文案
- 插件生态自带 UI
- 非核心实验性页面

这样可以在较小改动面下先验证方案是否顺畅。

## 10. 最终建议

结论如下：

- 当前项目 `ui/` 没有现成的多语言能力，属于英文硬编码为主的状态。
- 如果只追求最短时间接入，可以自研轻量方案；但从维护成本和演进空间看，不适合作为正式架构。
- 对当前 Paperclip UI，最合理的选择是 `i18next + react-i18next`。
- 实施方式应当是“先建基础设施，再逐页迁移”，而不是一次性全量重写。
- 默认语言建议保持英文，同时补上简体中文支持，为后续更多语言扩展留下明确结构。

## 11. 后续实施建议

如果进入实现阶段，建议下一份文档输出为实施计划，内容至少包括：

- 需要新增和修改的具体文件路径
- 第一批迁移页面清单
- 语言资源命名规范
- 测试策略
- 验收标准

这样可以直接指导后续代码改造，而不是停留在概念方案层。
